1. What We’re Building
We are building an RL environment for training LLMs to handle enterprise customer support. Unlike existing benchmarks that use scripted interactions, our environment is populated by generative agents — AI-driven customer and employee personas with persistent memory, personality traits, and evolving behavior.

The core idea combines two research threads:
	•	CoreCraft (arXiv 2601.09032, 2602.16179): An enterprise customer support RL environment with 2,500+ entities, 14 types, and 23 tools. Showed that high-fidelity environments produce agents that generalize to out-of-distribution benchmarks (25% → 37% pass rate on EnterpriseBench).
	•	Generative Agents (Park et al., UIST 2023): Simulated town of AI agents with memory streams, reflection, and planning that produce emergent believable behavior in a shared world.

Our synthesis: instead of scripted customers and static databases, the customers and employees are generative agents. They remember past interactions, have personalities, and their actions populate the world state organically. This means the CRM data, ticket histories, and order records are coherent byproducts of real interactions — not synthetic data.
1.1 Why This Matters
The judging criteria weights Environment Innovation at 40%. Our key differentiators are:
	•	Partially observable: The agent must infer customer emotional state, not read it from a field.
	•	Memory-driven coherence: A customer who called before remembers what happened. The agent sees real ticket history.
	•	Emergent world state: Databases are populated through agent interactions, not synthetic generation. This produces far more coherent and realistic data.
	•	Dynamic difficulty: Impatient customers, multi-issue tickets, and cross-agent handoffs create natural curriculum.

2.2 Key Design Decisions
	•	pi-mono as agent harness: Each generative agent (customer or employee) is a pi-mono instance running in RPC mode. pi-mono handles the LLM loop, tool execution, session persistence, and context management. We don’t reinvent any of this.
	•	RPC mode for orchestration: pi-mono’s RPC mode sends/receives JSON over stdin/stdout. Our Python orchestrator spawns pi processes and communicates via this protocol. This is language-agnostic and process-isolated.
	•	Folder-per-agent isolation: Each agent’s pi instance runs with cwd set to its own folder. pi’s built-in file tools (read/write/edit) are scoped to that directory. The agent’s memory, persona, and context live there.
	•	CLI tools over SQLite: All world-state mutations go through Python Click CLI commands. Agents call these via pi’s bash tool. The CLI layer enforces permissions (e.g., customers can only see their own orders).
	•	SQLite for shared world state: One world.db for entity data, one channels.db for the messaging system. Agents read/write through CLI tools, never directly.
	•	Organic data generation: Instead of seeding synthetic data, we run a “warm-up” phase where generative agents interact with each other. This populates the databases with coherent, realistic records.

3. pi-mono: The Agent Harness
3.1 What is pi-mono?
pi-mono (github.com/badlogic/pi-mono) is a TypeScript agent toolkit by Mario Zechner. It provides:
	•	pi-ai: Unified LLM API supporting Anthropic, OpenAI, Google, Groq, and any OpenAI-compatible endpoint.
	•	pi-agent-core: Agent runtime with tool calling, event-driven architecture, and message queuing.
	•	pi-coding-agent: Full agent CLI with file tools (read/write/edit/bash), session persistence, extensions, and skills.
	•	RPC mode: JSON over stdin/stdout. A parent process sends commands and receives events. This is how we orchestrate from Python.
	•	Extensions: Register custom tools and event handlers. This is how we wire in our CLI tools.
	•	Session persistence: JSONL-based, stored in .pi/ per working directory. This is agent memory.
3.2 RPC Protocol
Each agent process is started with:
pi --mode rpc --provider <provider> --model <model>
Communication is LF-delimited JSONL on stdin (commands) and stdout (events). Key commands:

Direction
Type
Example
stdin → pi
user_message
{"type":"user_message","content":"Customer says: My order is late"}
pi → stdout
tool_call
{"type":"tool_call","name":"bash","args":{"cmd":"check_order..."}}
pi → stdout
assistant_message
{"type":"assistant_message","content":"I see your order..."}
pi → stdout
turn_complete
{"type":"turn_complete"}

The Python orchestrator (agent_manager.py) wraps this in a class:
class PiAgent:
    def __init__(self, agent_id: str, agent_dir: Path, provider: str, model: str):
        self.proc = subprocess.Popen(
            ["pi", "--mode", "rpc", "--provider", provider, "--model", model],
            cwd=str(agent_dir),
            stdin=subprocess.PIPE, stdout=subprocess.PIPE, stderr=subprocess.PIPE
        )

    def send_message(self, content: str) -> str:
        cmd = json.dumps({"type": "user_message", "content": content})
        self.proc.stdin.write(cmd + "\n")
        self.proc.stdin.flush()
        # Read events until turn_complete
        response = ""
        while True:
            line = self.proc.stdout.readline()
            event = json.loads(line)
            if event["type"] == "assistant_message":
                response += event["content"]
            elif event["type"] == "turn_complete":
                break
        return response
3.3 Extensions (Custom Tools)
Each agent folder has an extensions/ directory. pi auto-discovers these on startup. Extensions register tools that the LLM can call. We use this to give different agents different tool sets.

Example extension for a customer agent (agents/customer_001/extensions/customer_tools.ts):
export default function(pi) {
  pi.registerTool({
    name: "raise_ticket",
    description: "Raise a new support ticket for an issue you are experiencing",
    parameters: {
      subject: { type: "string", description: "Brief subject line" },
      description: { type: "string", description: "Detailed description of the issue" }
    },
    execute: async (args) => {
      // Calls the CLI tool via bash, passing the agent ID for auth
      const result = await pi.tools.bash.execute({
        cmd: `gencraft-cli raise-ticket --agent-id customer_001 \
              --subject "${args.subject}" --description "${args.description}"`
      });
      return result;
    }
  });

  // Similarly: reply_ticket, check_my_orders, check_ticket_status
}
3.4 Agent Memory
pi-mono stores sessions as JSONL files in .pi/agent/sessions/ within each agent’s folder. This gives every agent persistent memory across episodes. When an agent is re-spawned, it can resume from where it left off.
Additionally, each agent has a memory/ folder for structured files:
	•	memory/interactions.jsonl — timestamped log of all interactions (who said what, tool results)
	•	memory/reflections.md — periodic self-written summaries (e.g., “Customer 001 has called 3 times about the desk issue and is getting frustrated”)
	•	memory/facts.json — key-value store of learned facts (e.g., {"preferred_contact": "email", "vip_status": true})
The agent can read and write these files using pi’s built-in read and write tools. The reflections mechanism is inspired by Generative Agents — after every N interactions, the agent is prompted to write a reflection summarizing what it has learned.

5.3 Notification Flow
When certain events happen, the orchestrator injects system messages into channels:
	•	New ticket assigned: System posts to #support: “Ticket #42 assigned to employee_support_01. Customer: Sarah Chen. Subject: Damaged desk.”
	•	Escalation: System posts to #escalations: “Ticket #42 escalated by employee_support_01. Reason: Refund exceeds $200 limit.”
	•	Customer reply: System posts to #support: “Customer replied on Ticket #42.”
Employee agents check their messages at the start of each turn. This creates realistic async communication patterns.

7. OpenEnv Integration
This is how we expose everything as an OpenEnv environment. The Training LLM (the model being post-trained) interacts with this interface. The generative agents run behind it.
7.1 Data Models
# models.py
from dataclasses import dataclass
from openenv.core.models import Action, Observation

@dataclass
class SupportAction(Action):
    tool_name: str      # One of the 10 employee tools
    tool_args: dict      # Tool-specific arguments

@dataclass
class SupportObservation(Observation):
    customer_message: str    # What the customer just said (or empty)
    tool_result: str         # Result of last tool call (or empty)
    ticket_context: str      # Current ticket summary
    internal_messages: str   # Any new Slack messages for this agent
    reward: float
    done: bool
    info: dict               # Metadata: step_count, satisfaction, etc.
7.2 Environment Implementation
# server/customer_support_env.py
from openenv.core.environment import Environment

class CustomerSupportEnv(Environment):

    async def reset(self) -> Observation:
        # 1. Pick a customer agent (or create a new one)
        # 2. Generate a ticket scenario:
        #    - Customer agent reads its life_context/ files
        #    - Customer agent decides on an issue to raise
        #    - Customer calls raise_ticket via CLI
        # 3. Assign ticket to the training agent
        # 4. Get customer's opening message
        # 5. Return initial SupportObservation with:
        #    - customer_message = opening complaint
        #    - ticket_context = ticket summary from DB
        #    - reward = 0.0, done = False
        pass

    async def step(self, action: SupportAction) -> StepResult:
        # 1. Execute tool_name(tool_args) against world.db via CLI
        # 2. If tool is 'send_reply':
        #    a. Feed the reply to the customer's pi-mono instance
        #    b. Customer agent generates response using:
        #       - Its persona + memory
        #       - The reply content
        #       - Its satisfaction level
        #    c. Customer may: reply, provide info, express frustration,
        #       say 'resolved', or escalate
        # 3. If tool is 'escalate':
        #    a. Post to #escalations channel
        #    b. Manager pi-mono instance processes
        #    c. Manager response fed back as internal_messages
        # 4. Compute reward (see Section 8)
        # 5. Check done conditions:
        #    - Customer says resolved -> done=True
        #    - Max steps exceeded -> done=True
        #    - Customer abandons (patience exhausted) -> done=True
        # 6. Update agent memory files
        # 7. Return SupportObservation
        pass

    def state(self) -> State:
        return {
            'episode_id': self.current_episode_id,
            'step_count': self.step_count,
            'customer_satisfaction': self.satisfaction,
            'ticket_status': self.current_ticket_status,
        }

8. Reward Function
The reward function is a weighted combination of four signals. It’s computed at every step, with the final reward applied when done=True.
8.1 Components
Component
Weight
Range
How It’s Measured
Task Resolution
0.40
0.0 or 1.0
Binary: did the customer say their issue is resolved?
Customer Satisfaction
0.30
0.0 – 1.0
Tracked per-step based on agent actions (see below)
Efficiency
0.15
0.0 – 1.0
1.0 if resolved in ≤5 steps, decays 0.1 per step after
Tool Appropriateness
0.15
0.0 – 1.0
Penalty for wrong tools, bonus for correct first-try
8.2 Satisfaction Tracker
The customer’s satisfaction starts at their satisfaction_baseline (from persona.json) and changes each step:
	•	+0.1: Agent uses correct tool on first try
	•	+0.15: Agent acknowledges customer’s frustration or previous interactions
	•	+0.2: Issue resolved
	•	-0.1: Agent uses wrong tool or irrelevant tool
	•	-0.15: Agent asks customer to repeat information already in ticket history
	•	-0.2: Agent ignores escalation signals
	•	-0.05: Each additional step (patience decay)
Satisfaction is clamped to [0.0, 1.0]. When it hits 0.0, the customer abandons the interaction (done=True).
8.3 Final Reward Formula
reward = (
    0.40 * resolution      # 1.0 if resolved, 0.0 if not
  + 0.30 * satisfaction    # final satisfaction score
  + 0.15 * efficiency      # max(0, 1.0 - 0.1 * max(0, steps - 5))
  + 0.15 * tool_score      # correct_tools / total_tool_calls

10. The Coherent World Argument
This is our key differentiator and the story we tell in the demo (Storytelling = 30% of judging). Here’s the argument:

10.1 The Problem with Synthetic Data
Most RL environments for customer support use synthetic databases — randomly generated customers, orders, and tickets. This creates incoherent data:
	•	A customer has 50 orders but no ticket history (unrealistic)
	•	Ticket notes reference events that never happened
	•	Order statuses don’t follow logical timelines
	•	The agent being trained learns to exploit patterns that only exist in synthetic data
10.2 Our Solution: Organic Data
When generative agents interact with each other through the system, every database record is a byproduct of a real interaction:
	•	Customer A calls about Order #1234 → ticket created with real context
	•	Agent writes notes to the ticket → notes reference actual conversation
	•	Customer A calls back → their memory includes the first interaction
	•	Agent sees the ticket history → it’s a real history, not synthetic

This means the training data is coherent by construction. The agent being trained sees a world that makes sense because it was built by agents that actually lived in it. This is the Generative Agents insight applied to enterprise RL.
10.3 Warm-Up Phase
Before training begins, we run a warm-up phase where generative agents interact freely for N episodes. This populates the databases. After warm-up, the world has realistic data: customers with histories, tickets with real resolutions, and a knowledge base that reflects actual issues encountered.
